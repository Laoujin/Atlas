> **TL;DR.** A Synology NAS *can* host [Vercel](https://vercel.com)-style per-PR preview deployments, but every layer has a sharp edge. Use **Cloudflare Tunnel + a wildcard hostname** [[32]](https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/do-more-with-tunnels/local-management/configuration-file/) for ingress (sidesteps NAT, CGNAT and wildcard certs in one move), **GHCR + SSH-and-`docker compose pull`** for delivery [[38]](https://www.tobyscott.dev/blog/2024/06/29/ghcr-synology-container-manager/) (Container Manager's UI can't auth to GHCR), and **a thin shell wrapper over Docker Compose** as the orchestrator — [Coolify](https://coolify.io)'s installer doesn't run on DSM [[58]](https://github.com/coollabsio/coolify/discussions/3166), [Dokploy](https://dokploy.com) breaks on DSM's old engine [[55]](https://github.com/Dokploy/dokploy/issues/3888). If your previews must be reachable by external reviewers on a public repo, **a $5/mo VPS or [Cloudflare Pages](https://pages.cloudflare.com) free tier is the better call** — reserve the NAS for storage and internal-only previews [[72]](https://www.simplehomelab.com/udms-02-hardware-nas-minipc-vps/).

## What "preview deployment" means

The pattern is uniform across PaaS vendors: a webhook fires on PR open or push, the platform builds the commit, and an ephemeral environment is published at a deterministic per-PR URL. [Netlify](https://www.netlify.com) popularised the term in 2016 [[10]](https://www.netlify.com/blog/2016/07/20/introducing-deploy-previews-in-netlify/) with `deploy-preview-{n}--{site}.netlify.app` [[1]](https://docs.netlify.com/deploy/deploy-types/deploy-previews/). Vercel [[2]](https://vercel.com/docs/deployments/environments), Cloudflare Pages [[3]](https://developers.cloudflare.com/pages/configuration/preview-deployments/), [Render](https://render.com) [[4]](https://render.com/docs/preview-environments) and [Fly.io](https://fly.io) [[5]](https://fly.io/docs/blueprints/review-apps-guide/) copy the same loop, with Render going further and cloning whole-stack databases per PR. Self-hosted clones — [Coolify](https://github.com/coollabsio/coolify) ⭐ 54k (Apr 2026) and [Dokploy](https://github.com/Dokploy/dokploy) ⭐ 34k (Apr 2026) — reproduce the invariants: one env per branch, unique URL, automatic teardown, status comment on the PR [[6]](https://coolify.io/docs/applications/ci-cd/github/preview-deploy) [[7]](https://docs.dokploy.com/docs/core/applications/preview-deployments). The point is shifting review left: a designer or PM clicks a URL, exercises the running build, and surfaces integration bugs while they are still cheap [[8]](https://www.okteto.com/blog/preview-environments-importance/) [[9]](https://gun.io/news/2025/11/preview-environments-early-feedback-no-surprises/).

## DSM 7.2 building blocks

DSM 7.2 reshapes Synology around five primitives that, together, make this realistic.

| Primitive | Purpose for previews | Sharp edge |
|---|---|---|
| **Container Manager** (replaces Docker pkg, 2023) | Compose stacks per PR via the Project tab [[11]](https://www.wundertech.net/container-manager-on-a-synology-nas/) [[48]](https://kb.synology.com/en-nz/DSM/help/ContainerManager/docker_project?version=7) | Docker engine 24.0.2, unsupported by upstream since June 2024 [[64]](https://www.xda-developers.com/synology-container-manager-outdated/); container ports/volumes locked post-creation, duplicate-to-modify [[12]](https://www.blackvoid.club/container-manager-24-0-2-1535/); x86_64 only — ARMv8 needs the [007revad community port](https://github.com/007revad/ContainerManager_for_all_armv8) ⭐ 208 [[13]](https://github.com/SynoCommunity/spksrc/issues/5667) [[14]](https://github.com/007revad/ContainerManager_for_all_armv8) |
| **Reverse Proxy** (Login Portal → Advanced) | Maps host:port → container [[15]](https://mariushosting.com/synology-how-to-use-reverse-proxy-on-dsm-7/) with one-click WebSocket headers [[16]](https://mariushosting.com/synology-some-docker-containers-need-websocket/) | Subdomain-only, no path routing, no wildcard upstream [[17]](https://community.synology.com/enu/forum/1/post/140686) — wildcard fan-out needs hand-rolled nginx |
| **Web Station** | Per-portal Nginx/Apache, HTTP/2, HSTS [[18]](https://www.synology.com/en-us/dsm/7.2/software_spec/web_station) [[19]](https://kb.synology.com/en-uk/DSM/help/WebStation/application_webserv_virtualhost?version=7) | DSM 7.2 silently dropped Apache `Options ExecCGI/FollowSymLinks` from vhosts [[20]](https://www.synoforum.com/threads/dsm-7-2-web-station-temporary-fix-for-apache-2-4-virtualhosts-losing-execcgi-and-other-options-directives.11156/) |
| **SSH + sudo** | CI deploy hook | Default home perms (777) silently break key auth — needs 755 / .ssh 755 / authorized_keys 644 [[21]](https://github.com/007revad/Synology_SSH_key_setup); only `administrators` may SSH; sudo NOPASSWD is a separate explicit step [[22]](https://www.synoforum.com/threads/passwordless-ssh.5889/) |
| **Task Scheduler** | Cron hook for cleanup/teardown [[23]](https://kb.synology.com/en-id/DSM/help/DSM/AdminCenter/system_taskscheduler?version=7) [[24]](https://www.damirscorner.com/blog/posts/20190301-SchedulingATaskOnASynologyNas.html) | Runs as root by default — single source of truth for periodic image-prune |
| **Btrfs snapshots** | Cheap per-preview rollback [[25]](https://www.synology.com/en-us/dsm/Btrfs) | Ext4-only J-series models lose this lever entirely [[26]](https://www.synology.com/en-us/dsm/feature/snapshot_replication) |

## DNS + TLS — the hard part

DSM's bundled [Let's Encrypt](https://letsencrypt.org) client only speaks HTTP-01, so it **cannot issue wildcard certs and requires port 80 open to the internet** [[27]](https://lippertmarkus.com/2020/03/14/synology-le-dns-auto-renew/). Three working escapes:

| Approach | Setup | Trade-off |
|---|---|---|
| **[acme.sh](https://acme.sh) DNS-01 + DSM deploy hook** | `acme.sh --issue --dns dns_cf -d example.com -d "*.example.com"`, then `--deploy-hook synology_dsm` with `SYNO_Username/Password/Scheme/Port/Certificate/Create` env vars [[28]](https://codeberg.org/neilpang/acme.sh/wiki/Synology-NAS-Guide) [[29]](https://blog.prabir.me/posts/wildcard-certs-auto-renewal-in-synology-nas-with-dns-challenge-via-acme-sh/) | First deploy needs `--insecure` (DSM has no trusted cert yet) [[29]](https://blog.prabir.me/posts/wildcard-certs-auto-renewal-in-synology-nas-with-dns-challenge-via-acme-sh/); CSRF must be **enabled** in Control Panel > Security or the hook 403s silently [[35]](https://dr-b.io/post/Synology-DSM-7-with-Lets-Encrypt-and-DNS-Challenge) |
| **Cloudflare Tunnel** | `cloudflared` ingress with `hostname: "*.preview.example.com"` routing every subdomain to one local origin [[32]](https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/do-more-with-tunnels/local-management/configuration-file/) | TLS terminated at Cloudflare's edge — no port-forward, no wildcard cert problem; works with proxied wildcard records on all CF plans [[31]](https://developers.cloudflare.com/dns/manage-dns-records/reference/wildcard-dns-records/) |
| **[Caddy](https://caddyserver.com) + On-Demand TLS** | Caddy container provisions per-host certs at first TLS handshake [[33]](https://caddyserver.com/docs/automatic-https) | Must wire an `ask` endpoint to prevent abuse; Let's Encrypt rate-limits 50 certs/registered domain/week [[30]](https://letsencrypt.org/docs/rate-limits/) — fine for tens of open PRs, not hundreds |

**Cloudflare Tunnel is the cleanest path** — it eliminates NAT/CGNAT issues, port-forwarding, and the wildcard-cert problem in one configuration file. The [nschaper/homelab-domain-hosting](https://github.com/nschaper/homelab-domain-hosting) ⭐ 0 reference repo documents the exact wiring for Synology behind consumer ISPs (Xfinity, etc.) [[68]](https://github.com/nschaper/homelab-domain-hosting). For teams that prefer to avoid the shell client, a [Python wrapper](https://github.com/lippertmarkus/synology-le-dns-auto-renew) ⭐ 7 issues wildcard certs via DNS-01 and imports them through the DSM API as a scheduled task [[34]](https://github.com/lippertmarkus/synology-le-dns-auto-renew).

## CI/CD: GitHub Actions → Synology

Four delivery patterns, ordered by recommendation strength.

| Pattern | What CI does | Network needs | Verdict |
|---|---|---|---|
| **GHCR push + SSH `docker compose pull`** | `docker build && push ghcr.io/org/img:pr-N`; ssh to NAS; `docker compose -p pr-N pull && up -d` | SSH reachable (Tailscale node, CF Tunnel, or NAT) | ✓ Best fit. Container Manager's UI cannot auth to GHCR ('Registry returned bad result') so the pull must run via the underlying Docker CLI [[38]](https://www.tobyscott.dev/blog/2024/06/29/ghcr-synology-container-manager/) — but the daemon itself copes fine. Per-PR isolation is the compose project name. |
| **rsync over SSH** | [appleboy/ssh-action](https://github.com/appleboy/ssh-action) ⭐ 6.1k (Apr 2026) for arbitrary commands [[36]](https://github.com/appleboy/ssh-action); [easingthemes/ssh-deploy](https://github.com/easingthemes/ssh-deploy) ⭐ 1.3k (Apr 2026) for rsync [[37]](https://github.com/easingthemes/ssh-deploy) | Same | ✓ Fine for static-build previews (no container build). ⚠ ssh-deploy private key **must have no passphrase** because rsync's ssh cannot prompt [[37]](https://github.com/easingthemes/ssh-deploy). |
| **Self-hosted GitHub runner on DSM** | Runner pulls jobs and builds in-place; symlinks /var/run/docker.sock into /volume1/docker [[39]](https://www.damirscorner.com/blog/posts/20230414-RunGitHubActionsOnASynologyNas.html); a real-world write-up reports it works but only at org-runner level [[63]](https://medium.com/perspikapps/the-secret-life-of-your-synology-nas-from-storage-to-ci-cd-powerhouse-ef219a13f70c) | Outbound only | ⚠ **Private repos only.** GitHub explicitly states self-hosted runners "should almost never be used for public repositories" — any forker can run a malicious workflow that exfiltrates secrets and the GITHUB_TOKEN [[40]](https://docs.github.com/en/actions/reference/security/secure-use). The poutine `pr_runs_on_self_hosted` rule flags this exact pattern [[41]](https://boostsecurityio.github.io/poutine/rules/pr_runs_on_self_hosted/). |
| **Webhook / agent ([Portainer](https://www.portainer.io), [Watchtower](https://containrrr.dev/watchtower/))** | CI POSTs to `/api/stacks/webhooks/<uuid>` [[45]](https://docs.portainer.io/user/kubernetes/applications/webhooks) with `rollout-restart` toggle [[47]](https://oneuptime.com/blog/post/2026-03-20-portainer-webhooks-github-actions/view), or pushes the image tag and lets Watchtower poll GHCR every 300 s [[46]](https://containrrr.dev/watchtower/usage-overview/) | Webhook URL reachable (CF Tunnel + Zero-Trust service tokens) [[44]](https://dev.to/shape93/automating-docker-deployments-with-github-actions-cloudflare-tunnels-and-portainer-2phm) | ⚠ Portainer stack webhooks are **Business-Edition-only** [[45]](https://docs.portainer.io/user/kubernetes/applications/webhooks) [[50]](https://docs.portainer.io/user/docker/stacks/webhooks) [[52]](https://www.portainer.io/blog/portainer-community-edition-ce-vs-portainer-business-edition-be-whats-the-difference) — a [community sidecar](https://github.com/aklinker1/portainer-stack-webhook) ⭐ 3 fills the CE gap [[51]](https://github.com/aklinker1/portainer-stack-webhook). Watchtower's 5-min poll loses the per-PR lifecycle signal (no PR-close → teardown). |

For NAT traversal, the official [tailscale/github-action@v4](https://github.com/tailscale/tailscale) ⭐ 31k (Apr 2026) [[69]](https://github.com/tailscale/tailscale/issues/15434) ([Tailscale](https://tailscale.com)) brings up an ephemeral OAuth-tagged tailnet node inside the runner [[42]](https://tailscale.com/kb/1276/tailscale-github-action) and connects to DSM's built-in OpenSSH (Tailscale SSH itself is unsupported on Synology) [[43]](https://tailscale.com/docs/integrations/synology). Cloudflare Tunnel + Zero-Trust service tokens is the alternative — Shape93's 2024 walk-through has the `curl -H "CF-Access-Client-Id: $X" -H "CF-Access-Client-Secret: $Y"` plumbing [[44]](https://dev.to/shape93/automating-docker-deployments-with-github-actions-cloudflare-tunnels-and-portainer-2phm).

## Orchestration: which one fits

Stratified by how far you stray from DSM-native territory.

| Option | ⭐ Stars (Apr 2026) | Preview-deploy story | DSM compatibility |
|---|---|---|---|
| **Container Manager + Compose** | n/a (built-in) | Roll your own: `docker compose -p pr-N` per PR over SSH [[48]](https://kb.synology.com/en-nz/DSM/help/ContainerManager/docker_project?version=7). Thinnest, most reliable path. | ✓ Native. Third-party CLIs like [syno-docker](https://github.com/scttfrdmn/syno-docker) ⭐ 2 [[49]](https://github.com/scttfrdmn/syno-docker) paper over the GUI's opinions. |
| **Portainer CE** | n/a (community-edition docs [[53]](https://mariushosting.com/how-to-install-portainer-on-your-synology-nas/)) | Stacks API + webhooks are the right API surface for "redeploy this PR" [[45]](https://docs.portainer.io/user/kubernetes/applications/webhooks) [[47]](https://oneuptime.com/blog/post/2026-03-20-portainer-webhooks-github-actions/view) | ✓ Installs as a container [[53]](https://mariushosting.com/how-to-install-portainer-on-your-synology-nas/), but stack webhooks are **BE-only** [[50]](https://docs.portainer.io/user/docker/stacks/webhooks) [[52]](https://www.portainer.io/blog/portainer-community-edition-ce-vs-portainer-business-edition-be-whats-the-difference); the [aklinker1/portainer-stack-webhook](https://github.com/aklinker1/portainer-stack-webhook) ⭐ 3 sidecar [[51]](https://github.com/aklinker1/portainer-stack-webhook) is required for free auto-update. |
| **Dokploy** | ⭐ 34k [[54]](https://github.com/Dokploy/dokploy) | First-class GitHub-only PR previews with PR-comment URLs and label-based opt-in [[7]](https://docs.dokploy.com/docs/core/applications/preview-deployments) | ✗ Swarm-centric design [[56]](https://www.virtua.cloud/learn/en/concepts/coolify-vs-dokploy-self-hosted-paas) collides with DSM's older Docker daemon — `client version 1.53 too new, max 1.43` on a DS920+ since v0.28 [[55]](https://github.com/Dokploy/dokploy/issues/3888). |
| **Coolify v4** | ⭐ 54k [[57]](https://github.com/coollabsio/coolify) | Cleanest preview UX: `{{pr_id}}.{{domain}}` template + wildcard DNS [[6]](https://coolify.io/docs/applications/ci-cd/github/preview-deploy) | ✗ Installer fails on DSM with `grep: /etc/os-release: No such file or directory`; the maintainer-blessed workaround is **an Ubuntu VM under VMM** [[58]](https://github.com/coollabsio/coolify/discussions/3166), at which point you've left DSM. |
| **[k3s](https://k3s.io)** | ⭐ 33k [[60]](https://github.com/k3s-io/k3s/issues/5080) | [Argo CD](https://argo-cd.readthedocs.io) ApplicationSet PR generator, one Application per open PR [[61]](https://argo-cd.readthedocs.io/en/latest/operator-manual/applicationset/Generators-Pull-Request/) | ✗ Bare metal is heroic: DSM ships without the overlay snapshotter kernel module, so containerd needs the native snapshotter [[59]](https://medium.com/@marco.mezzaro/k3s-on-synology-what-if-it-works-e980b4b09fcb), and even after the kernel-module hacks you still hit cgroup failures on small units like the DS220+ [[60]](https://github.com/k3s-io/k3s/issues/5080). The realistic homelab pattern is k3s in a VMM Linux VM with the Synology CSI plugin handling storage [[62]](https://geoffburkeblog.com/2024/02/home-lab-with-proxmox-synology-and-k3s/). |

**Pick:** Container Manager + Compose + a bash deploy script. Portainer CE if you want a stacks UI and accept the webhook sidecar [[51]](https://github.com/aklinker1/portainer-stack-webhook). Everything else either does not run on DSM or runs only via "give up on DSM and use a Linux VM" — at which point the NAS is not where the work happens.

## Nobody has published an end-to-end recipe

A targeted search across Reddit (r/homelab, r/synology, r/selfhosted), Hacker News, dev.to, Medium, and GitHub turns up **no end-to-end public write-up** of a working per-PR preview pipeline on a Synology NAS. The Synology + GitHub-Actions corpus stops well short of the goal: the most-linked tutorial pushes a single container over Tailscale to a webhook receiver with no notion of PR number, dynamic subdomain, or teardown [[73]](https://awesomeprogrammer.com/blog/2023/01/15/how-to-deploy-an-application-from-github-actions-to-your-synology-nas/); Damir's Corner stops at runner registration [[39]](https://www.damirscorner.com/blog/posts/20230414-RunGitHubActionsOnASynologyNas.html); the 2025 perspikapps "CI/CD powerhouse" piece covers builds and private deploys but never per-PR ephemeral environments [[63]](https://medium.com/perspikapps/the-secret-life-of-your-synology-nas-from-storage-to-ci-cd-powerhouse-ef219a13f70c). The closest off-the-shelf GitHub Action — [pullpreview/action](https://github.com/pullpreview/action) ⭐ 196 — supports exactly two providers (Lightsail and Hetzner) and explicitly does not target a generic SSH host [[74]](https://github.com/pullpreview/action/wiki/Providers) [[75]](https://pullpreview.com/) [[78]](https://dev.to/crohr/automated-preview-deployments-from-github-to-aws-in-10-minutes-17k1). When developers ask publicly how to recreate Vercel-style PR previews on infrastructure they own, the answers default to vendor PaaS — Static Web Apps, deployment slots — not to a self-hosted recipe [[76]](https://learn.microsoft.com/en-us/answers/questions/5593562/how-to-create-a-service-with-preview-mode-where-ea). Even purpose-built PR-preview projects like [neilhtennek/PreviewOps](https://github.com/neilhtennek/PreviewOps) ⭐ 46 frame the gap as "every team wants this, nobody wants to build it" and ship as managed services [[77]](https://github.com/neilhtennek/PreviewOps).

→ Anyone building this on a Synology in 2026 is doing original integration work. Plan for it.

## Real-world gotchas

The closest things to a partial Synology preview-deployment write-up are [nschaper/homelab-domain-hosting](https://github.com/nschaper/homelab-domain-hosting) ⭐ 0 (Cloudflare Tunnel + multiple subdomains, no PR lifecycle) [[68]](https://github.com/nschaper/homelab-domain-hosting) and the perspikapps self-hosted-runner post [[63]](https://medium.com/perspikapps/the-secret-life-of-your-synology-nas-from-storage-to-ci-cd-powerhouse-ef219a13f70c). What turns up in spades is operational pain.

- **Engine drift.** Container Manager's 2025 release ships Docker 24.0.2, unsupported by upstream since June 2024; users report permission and resource-monitoring bugs and recommend migrating to Portainer/Dockge/Dockhand instead [[64]](https://www.xda-developers.com/synology-container-manager-outdated/).
- **DSM updates wipe state.** The DSM 7.3 → 7.3.1 update silently deletes downloaded kernel modules; [telnetdoogie/synology-docker](https://github.com/telnetdoogie/synology-docker) ⭐ 150 publishes the iptables-modules-reinstall recipe [[65]](https://github.com/telnetdoogie/synology-docker/issues/41). nginx.conf is regenerated from `.mustache` templates on every update — hand-edited vhosts are wiped unless they live in `/usr/local/etc/nginx/sites-enabled` (invisible to the UI but survives) [[67]](https://www.synoforum.com/threads/nginx-custom-reverse-proxy-configuration-issue.1896/).
- **BTRFS snapshots eat the volume.** One operator hit 4.7 TB of accumulated snapshots in `@docker` on a 10.5 TB volume, and aggressive cleanup broke every container on the box [[66]](https://www.synoforum.com/threads/how-can-i-clear-out-a-massive-amount-of-btrfs-snapshots-i-am-running-out-of-space-fast-and-dont-know-whats-creating-these.9074/). Quota the snapshots, do not ignore them.
- **Tailscale Funnel ≠ wildcard ingress.** No wildcard support, one domain per device [[69]](https://github.com/tailscale/tailscale/issues/15434) — if you are not using Cloudflare Tunnel, reverse-proxy a single funnel hostname through DSM's reverse proxy.
- **Disk hygiene is not optional.** A `docker system prune` script wired into Task Scheduler is canon; mariushosting publishes the standard recipe [[70]](https://mariushosting.com/synology-how-to-clean-docker-automatically/), but it does not quantify reclamation, so monitor.
- **[Plex](https://www.plex.tv) co-tenancy got worse in 2025.** Synology removed the i915 graphics driver from DSM, "effectively disabling hardware transcoding on the Intel Celeron J4125 CPU" — both H.264 and H.265 transcoding are blocked at the kernel-driver level, shifting load onto CPU [[71]](https://nascompares.com/2025/09/24/how-to-get-hardware-transcoding-back-on-your-synology-nas/). If Plex shares the box with previews, Plex now wins the CPU race during family movie night.

## When NOT to do this on a NAS

The honest answer: when previews must be reachable by external reviewers, on a public repo, with reliable uptime — **don't**. A $5/mo VPS or Cloudflare Pages free tier is structurally better suited [[72]](https://www.simplehomelab.com/udms-02-hardware-nas-minipc-vps/). Reserve the Synology for storage, backup, and **internal-only** previews where the team is on the tailnet. The simplehomelab decision is blunt about the split: NAS for storage, mini PC for Docker, VPS for things that have to live on the public internet [[72]](https://www.simplehomelab.com/udms-02-hardware-nas-minipc-vps/).
